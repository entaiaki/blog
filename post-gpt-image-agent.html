Local GPT-Image Agent：给 ComfyUI 加一层生产级后端

2026年6月 · 预计阅读 8 分钟

ComfyUI 能跑 Flux 模型，但它的 HTTP API 是同步的 — 请求发过去，等生成完才返回。Agent 调用的时候动不动就超时。我决定给它加一层"后端服务"。

背景：为什么需要这层桥

我的 MCP 工具 generate_flux_image 调用 ComfyUI 的 API 生成图片。Agent 通过 MCP 调用它，但 ComfyUI 的 /prompt 接口是同步阻塞的：一张 1024×1280 的图跑 24 steps 需要 40-60 秒。Agent 的 HTTP 超时设短了就会断，设长了用户体验差。

需要一个中间层：Agent → MCP → Bridge Service → ComfyUI。Bridge 立刻返回任务 ID，Agent 轮询等结果。

架构概览

┌─────────┐     MCP      ┌──────────────┐    HTTP     ┌──────────┐
│  Agent  │ ──────────→  │  Bridge      │ ─────────→  │ ComfyUI  │
│         │ ←─ poll ──── │  (7861)      │ ←─ result ─ │ (8186)   │
└─────────┘              │              │             └──────────┘
                         │ Task Queue   │
                         │ Task Store   │
                         │ Worker Pool  │
                         └──────────────┘

核心文件 local_image_bridge_v03.py，依赖模块全部拆分到独立文件。

分阶段实现

P0 — 异步任务模型（6月10日）

最核心的功能：把同步调用拆成提交 + 轮询。

  • POST /submit — 提交任务，立刻返回 task_id
  • GET /task/{task_id} — 查询状态（PENDING → RUNNING → SUCCEEDED / FAILED）
  • GET /metrics — 队列大小、成功/失败计数
  • 保留旧的 POST /generate-image 同步端点作兼容
  • 任务事件持久化到 bridge_data/tasks.jsonl（JSONL 格式，方便事后分析）

新增文件：bridge_logging.py、bridge_task_store.py、bridge_worker.py。

P1 — 稳定性与路由（6月10-11日）

异步骨架跑通后，加生产环境必要的东西：

  • API Key 认证：可选，在 bridge_config.json 里配 api_key，客户端带 X-API-Key 头
  • 令牌桶限流：每个 API Key 独立限流，超限返回 429
  • Workflow 路由：请求里传 profile（如 "default"），配置文件决定用哪个 workflow JSON
  • 多 Worker：--workers N 启动 N 个后台线程并行处理
  • MCP 工具的 generate_flux_image 改为走 Bridge 异步模型

新增文件：bridge_security.py、bridge_workflow_router.py、bridge_client.py 等。

P2 — 运维与量化（6月11日）

最后一个阶段，让系统"可观测、可度量"：

  • Prometheus 指标：GET /metrics 输出 text/plain 格式，含计数器 + 延迟直方图
  • 任务取消：POST /task/{id}/cancel，PENDING 直接取消，RUNNING 尽最大努力中断
  • 超时 + 重试：per-task task_timeout_seconds，max_retries + 指数退避，每次重试记录 __attempt
  • 审计字段：每次请求记录 workflow_sha256、comfyui_version、bridge_version
  • 压测报告：6 请求 / 2 并发 / 512×512 / 6 steps，100% 成功率，avg 43.6s / p50 42.2s / p95 57.4s

遇到的一个坑：timeout 被覆盖

P2 实现重试逻辑时，发现 task_timeout_seconds=1 传进去后实际变成了 900。排查后发现是 bridge_job.py 里有个兜底默认值 900s，覆盖了请求里传的值。改成"请求传了就优先用请求的"后修好。 

# bridge_job.py 修前
task_timeout = max(job.timeout, 900)

# 修后
task_timeout = job.timeout if job.timeout else 900

这个 bug 教会我一件事：默认值永远应该是最后一道防线，不应该覆盖显式传入的值。

学到了什么

  1. 异步化不复杂：任务队列 + Worker 线程 + 状态轮询，几十行 Python 就能搭起来
  2. JSONL 比数据库好用：几十条任务记录不需要 PostgreSQL，json.dumps() + f.write() 足够，事后 grep | jq 分析
  3. 限流从第一天就要加：ComfyUI 本身不支持排队，Bridge 不做限流的话一个 Agent 就能把它打挂
  4. 压测要早做：load_test.py 帮我发现了 p95 延迟远高于 avg，说明队列堆积时尾部延迟会爆炸
  5. Dockerfile 写到项目里：哪怕现在用不上，它强迫你想清楚所有环境依赖
